## 1. 成果物サマリ

本アプリは、外食チェーンの購買担当者が仕入先と価格交渉する際に、判断材料が散在し交渉準備に時間がかかる、交渉結果が個人の記憶や紙メモに留まり次回に活かされない、という課題を解決するために開発しました。要件定義書が定める次の3つの状態を、MVP のスコープで実現しています。

1. **交渉前の情報一元化**: 相場情報・過去の交渉経緯・自社の購買計画を、交渉準備の前に1画面で確認できます。
2. **3ライン作戦シート + AI 生成**: 目標ライン・妥協ライン・撤退ラインの3段階（3ライン）を計算式で自動算出し、交渉のポイントや想定シナリオを AI が整理して提示します。**最終的な判断は人が行い、AI は判断材料の整理のみを担う設計**です。
3. **結果記録と判断継承ループ（BR-10）**: 交渉結果（決着単価・決着理由・所感）を構造化して記録し、次回以降の同一商材×取引先の交渉時に自動的に参照できます。

### 画面構成（①〜⑤）

| 画面 | 内容 |
| --- | --- |
| ログイン画面 | Google 認証（後述「4. 確定仕様・制約」参照） |
| ① 案件作成／案件一覧 | 交渉案件の新規作成・検索・一覧表示（企業・商材・ステータス等で絞り込み） |
| ② 情報収集 | 相場情報・過去の交渉経緯（自動参照）・自社計画を3カラムで同時に確認 |
| ③ 3ライン算出 | 目標／妥協／撤退の3ラインを自動算出し、手修正・修正理由の記録も可能 |
| ④ 作戦シート | 3ライン＋AI が生成した交渉ポイント・想定シナリオを表示（引用元の明示付き） |
| ⑤ 結果記録 | 決着単価・決着理由（構造化タグ）・担当所感を記録し、次回交渉へ引き継ぐ |

②〜⑤は「案件ワークスペース」として1つの画面群にまとまっており、現在どの段階にいるかが常に分かる作りになっています。

### 主要 API（本体アプリ、約23 API）

案件管理（`cases`）、相場情報（`rates`）、自社計画（`plans`）、3ライン算出（`lines`）、作戦シート生成（`strategy`）、結果記録（`results`）、過去経緯検索（`search`）、認証（`auth`）、ヘルスチェック（`health`）の各領域に分かれています。本体アプリ（CORE）と、過去経緯参照を担う知識検索エンジン（KRE、後述）は、内部の契約（インターフェース）を介してのみやり取りする構成にしており、どちらか一方の内部実装を変更してももう一方に影響しません。

## 3. 品質保証の実績

QA担当（シナモロール）による統合テスト・最終判定の結果です。

### テスト規模

- **テスト件数: 163件**（単体テスト・契約テスト・統合テストの合計）

### E2E（実際の画面操作に近い形での）検証結果

- **検索応答時間: 0.15〜0.44秒**（実測。要件定義書の非機能要件が定める「検索応答 目標3秒以内」を十分に満たしています）
- **AI 生成結果の検証**: 作戦シートの AI 生成において、価格・数値の「捏造」（実在しない相場や過去実績をあたかも根拠があるかのように示すこと）が発生しないことを検証済み（ゼロ件）
- **テナント越境の防止**: 他社（他テナント）のデータが混入しないことを検証し、越境を試みた場合は `403`（アクセス拒否）が返ることを確認
- **冪等性・監査ログ**: 同じ操作を誤って複数回実行してもデータが重複・破損しないこと（冪等性）、および操作履歴が監査ログとして記録されることを確認

### レビュー履歴（多段ゲート）

本 MVP は、実装が終わってから一度に検収するのではなく、次の順で段階的に品質を確認しながら開発を進めました。

1. **設計レビュー**（アーキテクト担当によるアーキテクチャ・調整シーム設計の確認）
2. **実装レビュー**（各担当実装後のコードレビュー）
3. **統合テスト**（QA担当による本体アプリ・KRE を通した約23 API・9テーブルの結合確認、契約テスト、テナント分離のセキュリティレビュー）
4. **再判定**（指摘事項の解消を確認したうえでの最終合格判定）

この多段ゲートにより、実装の後戻りコストを抑えつつ品質を積み上げています。

## 4. 検収時に明示する確定仕様・制約

検収にあたり、発注元に明示的にご確認いただきたい確定仕様・制約です。

### 認証は Google 認証（正式仕様）

- 本 MVP の認証方式は **Google 認証（Google Identity Services, GIS）** です。統括承認済みの正式仕様として実装・実機確認済みです。
- 当初検討していた Microsoft Entra 認証への移行が将来必要になった場合も、認証方式を切り替える仕組み（認証シーム `AUTH_MODE`）をあらかじめ実装しているため、**アプリ本体の作り直しなしに切り替え可能**です。

### KRE（AI 検索）の索引同期には反映遅延があります

- 過去の交渉経緯のうち、**データベースに保存済みの過去経緯（データベース由来のもの）は即座に参照可能**です。
- 一方、AI 検索の索引（インデックス）とナレッジグラフは、`build_index`（索引再構築処理）を再実行するまで新しいデータが反映されません。**リアルタイムに自動反映されるわけではない**点にご留意ください。
- 将来、この反映を自動化・高速化する仕組み（`retrieval_config.sync` の設定切り替え）をあらかじめ用意しており、設定変更のみで対応できる設計です。

### テナント（会社・組織）ごとのデータベース分離は未実装です

- 本 MVP は、1つのデータベースの中でデータを会社ごとに区別する「アプリ層でのフィルタリング」＋「AI 検索側のフィルタ」という**二層防御**で、他社データの越境ゼロを担保しています。
- 会社ごとに物理的に異なるデータベースを用意する方式（per-tenant-DB ルーティング）は、MVP の時点では未実装です。
- MVP は実質的に1テナント（1組織）運用を前提としており、越境防止は「テナント越境ゼロ」を確認する自動テスト（契約テスト）で継続的に担保しています（3章参照）。

### 相場情報の CSV 取込・帳票 PDF 出力は画面からは行えません（将来機能）

- 相場情報の CSV 取込（農水省形式）および手入力について、**画面上のボタンからの取込操作は MVP のスコープに含まれておらず**、運用担当者がスクリプトを実行して取り込む方式です（FR-02）。
- 作戦シートの帳票（PDF 等）出力についても、要件定義書の MVP スコープ注記により **MVP では実装対象外**です（FR-07）。画面上での閲覧・DB 保存はできますが、印刷・PDF 化を伴う配布用の帳票出力は将来機能として残しています。

### 役割ごとの権限管理（ロールベース認可）の本格運用には設計作業が必要です

- 要件定義書には「担当者／管理者」といったロール（役割）による認可の要件がありますが、本格的な運用（画面・API 単位での権限制御の作り込み）には、ユーザー情報を管理する `users` テーブルの設計が必要です。
- 現時点では未着手のため、**次の設計トピックとして申し送り**します。

## 5. 未確定事項と調整シーム

発注元に確認いただきたい事項のうち、次の5点は**回答が未確定の状態**です。ただし、これらはすべて「調整シーム」（設定値やインターフェースの差し替えで対応できる仕組み）として実装済みのため、**回答を受領した後もアプリ本体のコードを改修することなく差し替え可能**です。これは本プロジェクトの設計上の特長です。

1. **3ライン算出式**: 目標／妥協／撤退の3ラインを計算する式は、発注元から正式な算出ロジックを受領できていないため、暫定式 v1 として実装し、版（バージョン）を管理できる形にしています。正式な式を受領後、版を切り替えるだけで反映できます。
2. **案件番号の採番規則**: 案件番号（例: No.500001）の採番ルールは、専用の仕組み（インターフェース）越しに差し替え可能です。
3. **規格グレードの運用**: 商品規格の「グレード」を自由入力にするか、あらかじめ定めた選択肢（マスタ）から選ぶ運用にするかは、設定切り替えで対応できます。
4. **業務範囲**: 現在は「購買（仕入）交渉」のみを対象としていますが、将来「販売交渉」等へ範囲を広げる場合も、区分値（enum）の拡張で吸収できる設計です。
5. **テナント分離の水準**: 4章で述べた分離水準（二層防御／per-tenant-DB 等）は、発注元の運用体制（何社・何組織で利用するか）が確定した段階で見直せるよう設計しています。

## 6. 今後の拡張タスク

MVP 後の拡張候補として、知識検索エンジン（KRE）・判断継承ループまわりの最適化タスクを整理した依頼書を用意しています。

- 参照先: [`docs/TASK_knowledge-graph-optimization.md`](./TASK_knowledge-graph-optimization.md)

概要（(a)〜(f)）:

| タスク | 内容 |
| --- | --- |
| (a) 検索チューニング | ハイブリッド検索（キーワード検索＋意味検索）のスコアリング調整 |
| (b) 索引・グラフの同期最適化 | 増分同期（差分だけを反映する仕組み）の実装 |
| (c) グラフ補完の品質向上 | 関連する過去案件・知識の補完精度向上 |
| (d) 契約テストの維持 | (a)〜(c)・(e) いずれの変更後も必ず満たすべき受け入れ条件（テスト green の維持） |
| (e) 相場高騰要因の推定アルゴリズム | 相場が高騰する要因を推定するアルゴリズムの導入・調整（算出アルゴリズム仕様の受領後に本格着手） |
| (f) AI 生成プロンプトの最適化 | 作戦シート生成に使う AI へのプロンプト（指示文）の改善 |

いずれも本体アプリと疎結合な領域のため、MVP の枠組みを崩さずに段階的に着手できます。

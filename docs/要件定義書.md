## 1. 背景と目的 [RFP]

外食業は売上の約3割が食材費であり、仕入れ交渉の巧拙が粗利に直結する。しかし交渉は
担当者の経験・暗黙知に依存し、判断根拠が組織に残らない。本システムは購買交渉を
仕組み化し、次の3つの状態を実現する。

1. 交渉前に必要情報（相場・過去経緯・自社計画）を一元的に確認できる
2. 標準的な交渉作戦シート（目標/着地/撤退の3ライン＋AI交渉シナリオ）を生成できる
3. 交渉後に結果と定性情報を構造化記録し、次回の同一商材×取引先案件で自動参照できる（判断継承ループ BR-10）

- KGI: 粗利率の向上（原価率低減）、営業利益率の向上
- KPI: 交渉結果の改善（見積からの改善値）、交渉準備時間の短縮、利用率・交渉結果記録率
- MVP判定: 10名程度の被験者による実測テスト（準備時間短縮と推奨度）

[HEARING] さらに、中小飲食店で汎用可能なプラットフォームとして発展させ、将来的には
外部データ＋店舗状況からの食材相場推定、使うほど精度が上がるFBループを実現する。

## 5. モジュール分割と外部委託範囲（関連知識検索エンジン / KRE）[決定]

本節はモジュール分割の決定（2026-07-09 ユーザー承認済み）を反映する。詳細スキーマ
（フィールド設計・DDL・ID規則・sync実装）はアーキテクチャ設計書 §3〜§4 が正本として持ち、
本書は「要約＋帰属＋受け入れ条件」レベルに留める。

### 5.1 KRE の定義・責務・保有資産

- **モジュール名**: 関連知識検索エンジン / Knowledge Retrieval Engine（略称 **KRE**）。
  ユーザーが Obsidian で運用する Wiki の query/graph アルゴリズムに相当する部分の業務切り出し。
- **切り出し境界**: **検索＋ナレッジグラフ＋整合の一式**を1モジュールとして外部委託する。
- **責務**: 「テナント＋クエリ（案件番号／自由文／属性）を受け取り、関連する過去案件・知識と、
  グラフ展開した文脈・引用元を返す」。本体アプリはこれを内部サービス契約越しに呼ぶ。
- **保有資産**: AI Search インデックス、ナレッジグラフ（nodes/edges。MVP は NetworkX＋JSON、
  将来 Cosmos）、整合(sync)処理、埋め込み生成パイプライン。

### 5.2 本体アプリ ⇄ KRE の帰属境界

| 領域 | 帰属 | 内容 |
|---|---|---|
| FR-03 過去経緯参照（全体） | **KRE（委託）** | AI Search 検索＋グラフ展開＋引用元の供給一式 |
| FR-08 交渉ポイント/シナリオ — 検索・グラフ文脈の供給 | **KRE（委託）** | `retrieve` 結果・`graph_context`・`citations` の生成 |
| FR-08 交渉ポイント/シナリオ — 最終プロンプト・生成呼び出し | **本体** | KRE の `graph_context`/`citations` を受け取り AI 生成 |
| AI Search 索引構築・検索 | **KRE（委託）** | 冪等投入・ハイブリッド検索・テナントフィルタ |
| GraphRAG（ノード/エッジ・隣接展開） | **KRE（委託）** | 同一取引先別商材・同一変動理由他社事例の補完 |
| 整合(sync)・埋め込み生成 | **KRE（委託）** | 索引とグラフの一貫性維持、埋め込みパイプライン |
| その他全FR・画面・認証・DB・3ライン算出 | **本体** | FR-01/02/04/05/07/08/09、UI、Google認証（GIS）、MySQL/アプリ層テナントフィルタ |

要点: **FR-03 全体**と **FR-08 の検索供給部**が KRE、**FR-08 の最終生成**は本体。
テナント越境防止の最終責任は本体（認証・アプリ層テナントフィルタ）にあり、KRE はテナント境界を跨がない結果のみ返す。

### 5.3 接続方式

- **同一 FastAPI プロセス内のサービス層インターフェース**（`RetrievalEngine` Protocol）
  ＋**設定ファイル `retrieval_config`** による接続とする。将来 HTTP 分離できる粒度を保つ。
- 本体はこの Python Protocol に対してコーディングし、実装（スタブ ⇄ 本実装）を
  DI（依存性注入）で差し替え可能にする。
- サービス契約（要約。詳細はアーキ設計書）:
  - `retrieve(RetrieveRequest) -> RetrieveResult`（`tenant_id`＋クエリ→ hits／graph_context／citations）
  - `index_upsert(IndexEvent) -> None`（取込契約。本体がビジネステーブル書込後にイベント発行、
    整合=索引・グラフの一貫性維持は KRE 責務）
  - `health() -> EngineHealth`（index_ready / graph_ready / last_sync_at / doc_count）

### 5.4 契約テスト＋スタブによる並行開発と受け入れ条件

- **並行開発の担保**: `RetrieveResult` の JSON Schema と代表 fixture（鶏もも肉／丸紅畜産の案件）を
  用意し、本体は**スタブ実装**で FR-03／FR-08供給 を並行開発する。
- **受け入れ条件（4点）** [決定]:
  1. 契約テストが green（RetrieveResult の JSON Schema 適合）
  2. `retrieval_config` の差し替えで検索挙動が実際に変化すること
  3. 代表クエリで「同一取引先の別商材」「同一変動理由の他社事例」がグラフ補完されること
  4. テナント越境ゼロ（他テナントの hit／ノードが混入しない）

### 5.5 retrieval_config（調整ノブ＝別担当者が本体に触れずチューニング）

設定ファイル（YAML/JSON）。Key Vault／環境変数で上書き可。`config_version` で版管理する
（N-07 の外部化要件の具体）。

- `search.hybrid_weight`（0..1。キーワード BM25 ⇔ 意味ベクトルの重み）
- `search.top_k` / `search.min_score`
